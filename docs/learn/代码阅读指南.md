# 文澜排版代码阅读指南

这份指南的目标不是替你解释每一行代码，而是帮你建立一条稳定的阅读路线：先理解项目如何跑起来，再理解数据如何流动，最后逐个模块深入。你可以按本文顺序一行行看代码，遇到看不懂的地方再回到对应章节定位。

## 1. 先建立项目全貌

这个项目当前可以分成三条线：

1. **前端界面**：`src/` 目录，React 负责编辑器、预览、复制、主题、设置等界面。
2. **Markdown 渲染**：`src/markdown/` 目录，MarkdownIt 和插件负责把 Markdown 转成公众号可用 HTML。
3. **桌面端能力**：`src-tauri/` 目录，Tauri/Rust 负责桌面打包、微信图片上传、本地配置保存、图片代理。

建议先看自己写的业务代码，暂时不要进入这些目录：

- `node_modules/`：第三方依赖源码，初期不要看。
- `src-tauri/target/`：Rust 编译产物，不是手写代码。
- `dist/`：前端构建产物，如果存在也先跳过。
- `markdown-nice-master/`：原 mdnice 项目副本，可作为参考资料，不是当前重写项目的主线。

## 2. 第一遍阅读目标

第一遍不要急着追所有细节，只回答下面 6 个问题：

1. 应用从哪里启动？
2. 页面由哪些核心组件组成？
3. Markdown 内容保存在哪里？
4. Markdown 如何变成预览 HTML？
5. 点击“复制到微信”时发生了什么？
6. 图片上传和微信图床配置走哪条链路？

只要这 6 个问题答清楚，后面逐行看代码就不会迷路。

## 3. 推荐阅读顺序

### 第 0 步：看项目脚本和依赖

先读：

- `package.json`

重点看：

- `scripts.dev`：启动 Vite 前端开发服务。
- `scripts.dev:web`：同样只启动 Vite 前端，保留作语义化别名。
- `scripts.build`：TypeScript 编译 + Vite 打包。
- `scripts.tauri`：进入 Tauri 桌面端开发/打包流程。

你需要先建立这个判断：

- 前端界面和 Markdown 渲染主要看 `src/`。
- 桌面端微信图床、配置保存、图片代理主要看 `src-tauri/`。
- 当前图片上传前端调用的是 Tauri 命令，不再经过 Express 后端。

### 第 1 步：看前端启动入口

按顺序读：

1. `src/main.tsx`
2. `src/App.tsx`
3. `src/styles/globals.css`

#### `src/main.tsx`

这是 React 入口，重点理解：

- `ReactDOM.createRoot(...)` 把 React 应用挂到 HTML 里的 `#root`。
- `<App />` 是整个业务界面的根组件。
- `globals.css` 是全局样式入口。

这里只需要知道：应用从这里进入 `App.tsx`。

#### `src/App.tsx`

这是第一遍最重要的文件之一。建议逐段看：

1. import 区：先看它依赖了哪些模块。
2. `useStore()`：内容和主题来自全局状态。
3. `handleUploadFile()`：上传图片的统一入口。
4. 默认内容加载 `useEffect()`：无草稿时加载 `content.md`。
5. 同步滚动 `useEffect()`：连接编辑器和预览区。
6. JSX 返回结构：顶部栏、编辑区、预览区、底部状态栏、设置弹窗。

读完后，你应该能画出这个结构：

```text
App
├─ Header
│  ├─ UploadButton
│  ├─ ThemeMenu
│  ├─ Settings button
│  └─ CopyButton
├─ Main
│  ├─ MarkdownEditor
│  └─ Preview
├─ Footer
└─ SettingsDialog
```

这是整个项目的“总装配图”。后续读任何组件，都可以回到这里看它是怎么被调用的。

### 第 2 步：看全局状态

读：

- `src/store/index.ts`

重点理解：

- 这个项目的核心状态很少：
  - `content`：当前 Markdown 文本。
  - `markdownThemeId`：当前主题 ID。
- `setContent()`：编辑器输入时更新内容。
- `setMarkdownTheme()`：切换主题时更新主题 ID。
- `persist(...)`：把状态持久化到浏览器本地存储。

这份代码很适合逐行看，因为它说明了项目为什么没有复杂状态管理：

```text
用户输入 Markdown
    ↓
setContent(content)
    ↓
Zustand store 更新
    ↓
App 重新渲染
    ↓
Preview 收到新 content
```

读这一部分时，重点不是 Zustand API 本身，而是确认：项目的数据源很集中。

---

### 第 3 步：看编辑器组件

读：

- `src/components/Editor/MarkdownEditor.tsx`

重点分 4 段看：

1. `MarkdownEditorHandle`：父组件可以调用编辑器的哪些能力。
2. `useImperativeHandle()`：把内部 CodeMirror 能力暴露给 `App.tsx`。
3. `extensions`：CodeMirror 启用 Markdown、自动换行、粘贴事件处理。
4. `<CodeMirror />`：真正渲染编辑器。

关键问题：为什么这里要用 `ref`？

因为 `App.tsx` 需要在图片上传成功后，把 `![](url)` 插入到当前光标位置。这个能力不是普通 props 能完成的，所以编辑器通过 ref 暴露：

- `insertAtCursor()`
- `getScroller()`
- `getTopLine()`
- `scrollToLine()`

其中后 3 个是同步滚动用的。

---

### 第 4 步：看预览组件

读：

- `src/components/Preview/Preview.tsx`

重点理解 3 件事：

1. 预览 HTML 来自 `render(content)`。
2. 主题 CSS 不是直接写在组件里，而是通过 `replaceStyle()` 注入到页面。
3. 微信图片预览会先走 `toProxyHtml()` 改写图片地址，解决微信图片防盗链问题。

核心链路是：

```text
content Markdown
    ↓
render(content)
    ↓
toProxyHtml(html)
    ↓
setHtml(...)
    ↓
dangerouslySetInnerHTML
    ↓
页面预览
```

读到 `dangerouslySetInnerHTML` 时要停一下：这是 React 里直接插入 HTML 的方式。这里使用它是合理的，因为 Markdown 渲染结果本来就是 HTML。

---

### 第 5 步：看 Markdown 渲染核心

按顺序读：

1. `src/markdown/parser.ts`
2. `src/markdown/data-line.ts`
3. `src/markdown/plugins/*.ts`

#### `src/markdown/parser.ts`

这是第二个最重要的文件。它决定 Markdown 最终变成什么 HTML。

建议按这个顺序读：

1. 第三方插件 import。
2. 自定义插件 import。
3. `highlight()` 代码高亮函数。
4. `new MarkdownIt(...)` 初始化。
5. `.use(...)` 插件链。
6. `render(markdown)` 导出函数。

最重要的是插件顺序。MarkdownIt 插件会改 token 或渲染结果，顺序不同，结果可能不同。

你可以把它理解成流水线：

```text
Markdown 文本
    ↓
MarkdownIt 解析成 token
    ↓
标题/表格/数学公式/脚注/TOC/图片/引用等插件依次处理
    ↓
生成 HTML
```

#### `src/markdown/data-line.ts`

这个文件服务于同步滚动。

它给顶层 Markdown 块加上：

```html
<section data-line="10">...</section>
```

这样预览区就知道某个 HTML 块来自 Markdown 源码的哪一行。

#### `src/markdown/plugins/*.ts`

插件建议最后看，因为它们是细节增强。可以按这个顺序：

1. `heading-span.ts`：标题结构增强。
2. `table-container.ts`：表格外层容器。
3. `math.ts`：数学公式。
4. `link-footnote.ts`：链接转脚注。
5. `list-item-wrap.ts`：列表项包裹。
6. `image-flow.ts`：图片展示增强。
7. `multi-quote.ts`：多级引用样式。
8. `imsize.ts`：图片尺寸语法。
9. `remove-pre.ts`：如果未被主链路使用，可以低优先级看。

看插件时不要一开始就钻 API。先问：这个插件输入什么 token，输出什么 HTML 或 token 变化。

---

### 第 6 步：看复制到微信

按顺序读：

1. `src/components/Copy/CopyButton.tsx`
2. `src/markdown/converter.ts`
3. `src/utils/clipboard.ts`
4. `src/utils/style.ts`

#### `CopyButton.tsx`

重点看：

```text
点击按钮
    ↓
solveHtml()
    ↓
copyHtml(html)
    ↓
显示复制成功/失败状态
```

#### `converter.ts`

这是复制功能的核心。

它做几件重要的事：

1. 找到预览区 DOM：`#nice-rich-text-box`。
2. 给顶层元素加 `data-tool`。
3. 把预览里的微信图片代理地址还原成原始 mmbiz 地址。
4. 删除同步滚动用的 `data-line`。
5. 对 MathJax 相关 HTML 做微信兼容处理。
6. 读取当前页面里的主题 CSS。
7. 用 `juice.inlineContent()` 把 CSS 转成 inline style。

为什么要 inline style？

因为微信公众号编辑器对外部 CSS、style 标签、复杂选择器支持不好，最稳的是把样式写到每个标签的 `style="..."` 上。

#### `clipboard.ts`

重点理解：

- 优先用现代 Clipboard API 写入 `text/html`。
- 不支持时回退到 `document.execCommand("copy")`。
- 必须写 `text/html`，否则粘贴到微信时只会变成普通文本或丢格式。

### 第 7 步：看主题系统

按顺序读：

1. `src/themes/index.ts`
2. `src/themes/basic.ts`
3. `src/themes/markdown/default.ts`
4. `src/themes/markdown/elegant.ts`
5. `src/themes/markdown/tech.ts`
6. `src/themes/code/*.ts`
7. `src/components/Theme/ThemeMenu.tsx`

重点理解主题被拆成两层：

1. `basic`：基础样式，永远存在。
2. markdown 主题 + code 主题：用户切换主题时替换。

主题切换链路：

```text
ThemeMenu 点击主题
    ↓
setMarkdownTheme(id)
    ↓
Zustand 保存 markdownThemeId
    ↓
Preview 监听 markdownThemeId
    ↓
replaceStyle() 替换 CSS
    ↓
预览区样式变化
```

这套设计比较轻量，适合当前项目：主题本质是 CSS 字符串，不需要复杂主题框架。

### 第 8 步：看图片上传链路

按顺序读前端：

1. `src/components/Upload/UploadButton.tsx`
2. `src/App.tsx` 里的 `handleUploadFile()`
3. `src/utils/upload.ts`
4. `src/components/Settings/SettingsDialog.tsx`

再读桌面端：

1. `src-tauri/src/lib.rs`
2. `src-tauri/src/config.rs`
3. `src-tauri/src/wechat.rs`

#### 前端上传链路

```text
点击上传图片 / 粘贴图片
    ↓
UploadButton 或 MarkdownEditor paste handler
    ↓
App.handleUploadFile(file)
    ↓
uploadImage(file)
    ↓
Tauri invoke("upload_image")
    ↓
Rust 上传到微信
    ↓
返回图片 URL
    ↓
insertAtCursor("![](url)")
```

#### `src/utils/upload.ts`

重点看：

- 文件类型限制：jpg/png/gif。
- 文件大小限制：10MB。
- `file.arrayBuffer()` 转字节数组。
- `invoke("upload_image", ...)` 调 Rust 命令。
- `NOT_CONFIGURED` 错误会被前端识别，用来打开设置弹窗。

#### `SettingsDialog.tsx`

重点看：

- 打开时调用 `get_config` 回显配置。
- 保存时调用 `save_config`。
- AppID/AppSecret 只保存在本机。

#### Rust 上传链路

`src-tauri/src/lib.rs` 是 Tauri 主注册点：

- 注册 `upload_image`。
- 注册 `get_config`。
- 注册 `save_config`。
- 注册 `wximg` 图片代理协议。

`src-tauri/src/config.rs` 负责：

- 从 Tauri app data 目录读取配置。
- 保存 `config.local.yaml`。
- 保存后清除微信 access_token 缓存。

`src-tauri/src/wechat.rs` 负责：

- 检查图片类型和大小。
- 获取并缓存微信 `access_token`。
- 调微信永久素材接口上传图片。
- token 失效时清缓存并重试一次。
- 代理拉取 mmbiz 图片，解决本地预览防盗链。

### 第 9 步：看同步滚动

按顺序读：

1. `src/App.tsx` 中创建同步滚动的 `useEffect()`。
2. `src/components/Editor/MarkdownEditor.tsx` 中的 `getTopLine()` 和 `scrollToLine()`。
3. `src/markdown/data-line.ts`。
4. `src/utils/syncScroll.ts`。

核心思路：

```text
Markdown 源码行号
    ↓
data-line 注入到预览 HTML
    ↓
编辑器滚动时拿到顶部行号
    ↓
根据 data-line 找预览区对应位置
    ↓
预览区滚动
```

反方向也一样：

```text
预览区 scrollTop
    ↓
根据 data-line 反推源码行号
    ↓
CodeMirror 滚到该行
```

`syncScroll.ts` 里最值得仔细读的是：

- `anchors()`：收集预览区所有 `data-line` 锚点。
- `syncEditorToPreview()`：编辑器滚动同步到预览。
- `syncPreviewToEditor()`：预览滚动同步到编辑器。
- `lockUntil`：防止两个区域互相触发滚动，形成来回抖动。

### 第 10 步：看微信图片代理

按顺序读：

1. `src/utils/imageProxy.ts`
2. `src-tauri/src/lib.rs` 的 `handle_wximg()`
3. `src-tauri/src/wechat.rs` 的 `fetch_proxied_image()`
4. `src/markdown/converter.ts` 的 `fromProxyHtml()` 调用

问题背景：微信 mmbiz 图片有防盗链，本地预览直接请求可能显示异常。

解决方案：

```text
渲染预览时
mmbiz 图片原始地址
    ↓
toProxyHtml()
    ↓
wximg://localhost/?url=...
    ↓
Tauri 自定义协议带 Referer 拉图
    ↓
本地预览正常显示
```

复制到微信前：

```text
代理地址
    ↓
fromProxyHtml()
    ↓
还原成原始 mmbiz 地址
    ↓
复制给微信公众号编辑器
```

这条链路要重点看，因为它解释了为什么预览和复制使用的图片地址不一样。

## 4. 一行行读代码时的建议方法

建议每个文件都按 5 层来读：

### 第 1 层：文件职责

先用一句话写下来：这个文件负责什么？

例如：

```text
src/markdown/converter.ts：把预览区 HTML 转成微信公众号可粘贴的最终 HTML。
```

如果一句话说不清，说明还没读懂，不要继续钻细节。

### 第 2 层：外部依赖

看 import：

- 来自 React 的是什么？
- 来自本项目其他文件的是什么？
- 来自第三方库的是什么？

import 区通常能告诉你这个文件处在什么层级。

### 第 3 层：对外暴露

看 export：

- 默认导出组件？
- 导出工具函数？
- 导出类型？

这能帮你判断：谁会调用它。

### 第 4 层：核心流程

把主流程写成箭头：

```text
输入
 ↓
处理
 ↓
输出
```

不要一开始就解释每个变量。

### 第 5 层：边界处理

最后再看：

- 错误怎么处理？
- 空数据怎么处理？
- 外部 API 失败怎么办？
- 是否有大小、类型、域名限制？

比如图片上传里，类型、大小、未配置、微信错误都属于边界处理。

## 9. 读代码时容易混淆的点

### 1. 预览 HTML 和复制 HTML 不是一回事

预览时：

- 需要在本地浏览器/WebView 里显示正常。
- 微信图片可能走代理。
- CSS 可以通过 style 标签注入。

复制时：

- 需要给微信公众号编辑器使用。
- 图片地址要还原成原始 mmbiz 链接。
- CSS 要内联到标签上。
- 同步滚动用的 `data-line` 要删掉。

所以 `Preview.tsx` 和 `converter.ts` 的职责不同。

### 2. 图片上传只有一条当前实现

- `src-tauri/src/wechat.rs`：当前实际使用的 Rust 实现。

历史文档里可能还会提到 Node/Express 实现，那是旧方案，不再对应当前代码目录。

### 3. Markdown 内容不在组件局部 state 里

内容在 Zustand store 里：

- `MarkdownEditor` 只是显示和编辑。
- `App` 把 `content` 传给编辑器和预览。
- `Preview` 根据 `content` 渲染。

### 4. 主题不是 React 组件主题系统

主题本质是 CSS 字符串：

- 预览时注入 style 标签。
- 复制时读取这些 CSS 并内联。

这种方式简单直接，适合文澜排版。

### 5. `data-line` 只为同步滚动服务

`data-line` 不应该复制到微信，所以 `converter.ts` 会删除它。

## 12. 建议你重点关注的可控性问题

既然这个项目是 AI 写出来的，后期想可控，重点不要只看“能不能跑”，还要看这些：

1. **状态是否集中**：当前主要集中在 `src/store/index.ts`，比较好控。
2. **外部 API 是否隔离**：微信接口集中在 `src-tauri/src/wechat.rs`。
3. **复制逻辑是否单点维护**：集中在 `src/markdown/converter.ts`。
4. **Markdown 扩展是否有边界**：集中在 `src/markdown/plugins/`。
5. **主题是否容易新增**：集中在 `src/themes/` 和 `src/themes/index.ts`。
6. **安全边界是否清楚**：图片类型、大小、代理域名、AppSecret 本地保存都要重点看。

## 13. 如果你要继续改代码，建议从小任务开始

适合练手的改动：

1. 新增一个 Markdown 主题。
2. 调整顶部工具栏按钮样式。

4. 给复制失败提示改成更明确的文案。

2. 微信上传接口。
3. Tauri 自定义协议。
4. 复制 HTML 清洗逻辑。
5. 同步滚动算法。

这些地方一改就可能影响核心链路，适合读懂后再动。
