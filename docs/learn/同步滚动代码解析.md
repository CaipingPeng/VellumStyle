# 同步滚动代码解析

这份文档专门解释当前项目里“编辑器 ↔ 预览区同步滚动”的实现。读完后，你应该能回答三个问题：

1. 为什么预览区知道 Markdown 源码的行号？
2. 编辑器滚动时，预览区怎么找到对应位置？
3. 预览区滚动时，编辑器怎么滚回对应源码行？

## 1. 相关文件总览

同步滚动涉及 5 个文件：

| 文件 | 作用 |
| --- | --- |
| `src/App.tsx` | 把编辑器、预览区、同步滚动工具接起来 |
| `src/components/Editor/MarkdownEditor.tsx` | 暴露编辑器滚动容器、当前顶部行号、滚动到指定行的方法 |
| `src/components/Preview/Preview.tsx` | 暴露预览区滚动容器 |
| `src/markdown/data-line.ts` | 给预览 HTML 顶层块注入 `data-line` 行号 |
| `src/utils/syncScroll.ts` | 同步滚动核心算法 |

整体关系是：

```text
Markdown 源码
    ↓
markdown-it 解析
    ↓
data-line 插件给预览 HTML 加行号
    ↓
Preview 显示带 data-line 的 HTML
    ↓
App 拿到 Editor 和 Preview 的滚动容器
    ↓
createScrollSync 建立双向滚动监听
```

## 2. 先看最终效果

项目目标是：

```text
左侧编辑器滚动
    ↓
右侧预览区滚到对应内容
```

同时也支持：

```text
右侧预览区滚动
    ↓
左侧编辑器滚到对应源码位置
```

这里的难点是：

- 编辑器是纯 Markdown 文本。
- 预览区是渲染后的 HTML。
- Markdown 的一行，渲染后高度可能完全不同。
- 有些 Markdown 块很高，比如图片、代码块、表格。
- 所以不能简单用滚动百分比同步。

当前项目采用的是 **源码行号锚点同步**。

## 3. 核心思路：用 `data-line` 建立映射

MarkdownIt 解析 Markdown 时，block token 自带源码行号范围：

```ts
token.map = [startLine, endLine]
```

当前项目利用这个信息，给预览 HTML 的顶层块加上：

```html
<section data-line="10">...</section>
```

这样预览区就知道：

```text
这个 HTML 块来自 Markdown 第 10 行附近
```

同步滚动时，不是按百分比，而是按这些行号锚点估算位置。

## 4. `data-line.ts`：给预览块加源码行号

文件：`src/markdown/data-line.ts`

代码核心：

```ts
export default function dataLinePlugin(md: MarkdownIt): void {
  md.core.ruler.push("inject_data_line", (state) => {
    for (const token of state.tokens) {
      if (token.level === 0 && token.map && token.nesting !== -1) {
        token.attrSet("data-line", String(token.map[0]));
      }
    }
    return true;
  });
}
```

逐句理解：

```ts
md.core.ruler.push("inject_data_line", ...)
```

把一个插件插到 MarkdownIt 的 core 阶段。

core 阶段可以遍历所有 token，适合做统一后处理。

---

```ts
for (const token of state.tokens)
```

遍历 MarkdownIt 解析出来的所有 token。

---

```ts
if (token.level === 0 && token.map && token.nesting !== -1)
```

只给一部分 token 加 `data-line`。

三个条件分别是：

| 条件 | 含义 |
| --- | --- |
| `token.level === 0` | 只处理顶层块，避免嵌套块太多 |
| `token.map` | token 必须有源码行号信息 |
| `token.nesting !== -1` | 不处理闭合 token |

为什么只处理顶层块？

因为如果给所有嵌套元素都加行号，预览区锚点会太密，滚动同步容易抖动，也会增加计算成本。

---

```ts
token.attrSet("data-line", String(token.map[0]));
```

把源码起始行写到 HTML 属性里。

`token.map[0]` 是起始行，0-based，也就是从 0 开始数。

例如 Markdown：

```md
# 标题

第一段文字
```

可能生成：

```html
<h1 data-line="0">标题</h1>
<p data-line="2">第一段文字</p>
```

## 7. `MarkdownEditor.tsx`：编辑器提供滚动能力

文件：`src/components/Editor/MarkdownEditor.tsx`

同步滚动相关接口：

```ts
export interface MarkdownEditorHandle {
  insertAtCursor: (text: string) => void;
  getScroller: () => HTMLElement | null;
  getTopLine: () => number;
  scrollToLine: (line: number) => void;
}
```

其中同步滚动用的是：

| 方法 | 用途 |
| --- | --- |
| `getScroller()` | 拿 CodeMirror 的滚动容器 |
| `getTopLine()` | 获取编辑器当前顶部可视行号 |
| `scrollToLine(line)` | 把编辑器滚动到指定行 |

### 7.1 `getScroller()`

代码：

```ts
getScroller: () => cmRef.current?.view?.scrollDOM ?? null,
```

CodeMirror 内部真正滚动的 DOM 是 `view.scrollDOM`。

所以这里把它暴露给外部。

### 7.2 `getTopLine()`

代码：

```ts
getTopLine: () => {
  const view = cmRef.current?.view;
  if (!view) {
    return 0;
  }
  const top = view.scrollDOM.scrollTop;
  const blockInfo = view.lineBlockAtHeight(top);
  return view.state.doc.lineAt(blockInfo.from).number - 1;
},
```

逐句看：

```ts
const view = cmRef.current?.view;
```

拿到 CodeMirror 实例。

---

```ts
const top = view.scrollDOM.scrollTop;
```

拿到编辑器当前滚动高度。

---

```ts
const blockInfo = view.lineBlockAtHeight(top);
```

问 CodeMirror：这个滚动高度对应哪一块文本行？

---

```ts
return view.state.doc.lineAt(blockInfo.from).number - 1;
```

CodeMirror 行号是 1-based，第一行是 1。

但是 `data-line` 使用的是 MarkdownIt 的 0-based 行号。

所以这里要减 1。

```text
CodeMirror 第 1 行 → data-line 0
CodeMirror 第 2 行 → data-line 1
CodeMirror 第 3 行 → data-line 2
```

### 7.3 `scrollToLine(line)`

代码：

```ts
scrollToLine: (line) => {
  const view = cmRef.current?.view;
  if (!view) {
    return;
  }
  const docLine = Math.min(Math.max(line + 1, 1), view.state.doc.lines);
  const pos = view.state.doc.line(docLine).from;
  view.dispatch({effects: EditorView.scrollIntoView(pos, {y: "start"})});
},
```

逐句看：

```ts
const docLine = Math.min(Math.max(line + 1, 1), view.state.doc.lines);
```

外部传入的是 0-based 行号。

CodeMirror 需要 1-based 行号。

所以先 `line + 1`。

然后用 `Math.max` 和 `Math.min` 限制范围，避免行号小于 1 或大于总行数。

---

```ts
const pos = view.state.doc.line(docLine).from;
```

拿到这一行起始位置。

---

```ts
view.dispatch({effects: EditorView.scrollIntoView(pos, {y: "start"})});
```

让 CodeMirror 滚动到这个位置，并尽量放到视口顶部。

## 8. `App.tsx`：把编辑器和预览区接起来

文件：`src/App.tsx`

相关 ref：

```ts
const editorRef = useRef<MarkdownEditorHandle>(null);
const previewRef = useRef<PreviewHandle>(null);
```

`App` 同时持有编辑器和预览区的 ref。

这样它能拿到：

```text
editorRef.current.getScroller()
editorRef.current.getTopLine()
editorRef.current.scrollToLine(line)
previewRef.current.getScroller()
```

### 8.1 创建同步滚动

代码：

```ts
useEffect(() => {
  let sync: {destroy: () => void} | null = null;
  let raf = 0;
  const attach = () => {
    const editor = editorRef.current;
    const preview = previewRef.current;
    const editorScroller = editor?.getScroller();
    const previewScroller = preview?.getScroller();
    if (!editor || !editorScroller || !previewScroller) {
      raf = requestAnimationFrame(attach);
      return;
    }
    sync = createScrollSync({
      editorScroller,
      previewScroller,
      getEditorTopLine: () => editor.getTopLine(),
      scrollEditorToLine: (line) => editor.scrollToLine(line),
    });
  };
  attach();
  return () => {
    cancelAnimationFrame(raf);
    sync?.destroy();
  };
}, []);
```

这段是同步滚动的装配层。

### 8.2 为什么要 requestAnimationFrame 重试？

这段：

```ts
if (!editor || !editorScroller || !previewScroller) {
  raf = requestAnimationFrame(attach);
  return;
}
```

原因是：React 首次渲染时，组件 ref 不一定立刻可用；CodeMirror 的内部 `.cm-scroller` 也可能第一帧还没挂载。

所以这里不是失败就放弃，而是下一帧继续尝试。

直到拿到：

- 编辑器实例
- 编辑器滚动 DOM
- 预览区滚动 DOM

才真正调用：

```ts
createScrollSync(...)
```

### 8.3 为什么 cleanup 要 destroy？

```ts
return () => {
  cancelAnimationFrame(raf);
  sync?.destroy();
};
```

组件卸载时要清理：

- 未完成的 `requestAnimationFrame`
- 已经绑定的 scroll 事件监听

否则会出现内存泄漏，或者组件卸载后还在响应滚动。

## 9. `syncScroll.ts`：核心同步算法

文件：`src/utils/syncScroll.ts`

这是最值得精读的文件。

### 9.1 输入参数

```ts
export interface ScrollSyncOptions {
  editorScroller: HTMLElement;
  previewScroller: HTMLElement;
  getEditorTopLine: () => number;
  scrollEditorToLine: (line: number) => void;
}
```

它不直接依赖 React，也不直接依赖 CodeMirror。

它只需要四样东西：

| 参数 | 作用 |
| --- | --- |
| `editorScroller` | 编辑器滚动 DOM |
| `previewScroller` | 预览区滚动 DOM |
| `getEditorTopLine` | 获取编辑器当前顶部行号 |
| `scrollEditorToLine` | 让编辑器滚到某一行 |

这样设计的好处是：同步算法和具体组件解耦。

### 9.2 防抖动锁：`lockUntil`

代码：

```ts
const LOCK_MS = 80;
```

```ts
let lockUntil = 0;
```

为什么需要锁？

因为同步滚动是双向的。

如果没有锁，会发生：

```text
用户滚动编辑器
    ↓
程序滚动预览区
    ↓
预览区触发 scroll 事件
    ↓
程序又滚动编辑器
    ↓
编辑器再次触发 scroll 事件
    ↓
来回互相触发
```

所以每次程序主动滚动另一区域时，会设置：

```ts
lockUntil = Date.now() + LOCK_MS;
```

在这 80ms 内，另一区域触发的 scroll 事件会被忽略。

### 9.3 `anchors()`：收集预览区锚点

代码：

```ts
const anchors = (): {line: number; top: number}[] => {
  const els = previewScroller.querySelectorAll<HTMLElement>("[data-line]");
  const list: {line: number; top: number}[] = [];
  for (const el of els) {
    const line = Number(el.getAttribute("data-line"));
    if (!Number.isNaN(line)) {
      list.push({line, top: el.offsetTop});
    }
  }
  list.sort((a, b) => a.line - b.line);
  return list;
};
```

这个函数把预览区所有带 `data-line` 的元素收集成数组。

每个元素变成：

```ts
{ line: 10, top: 320 }
```

含义是：

```text
Markdown 第 10 行对应的预览块，距离预览容器顶部 320px
```

这里用的是：

```ts
el.offsetTop
```

也就是该元素在预览区里的垂直位置。

最后排序：

```ts
list.sort((a, b) => a.line - b.line);
```

保证锚点按源码行号升序排列。

## 10. 编辑器滚动 → 预览滚动

函数：`syncEditorToPreview()`

代码：

```ts
const syncEditorToPreview = () => {
  const list = anchors();
  if (list.length === 0) {
    return;
  }
  const line = getEditorTopLine();
  let prev = list[0];
  let next = list[list.length - 1];
  for (let i = 0; i < list.length; i++) {
    if (list[i].line <= line) {
      prev = list[i];
      next = list[i + 1] ?? list[i];
    } else {
      break;
    }
  }
  let top: number;
  if (next.line === prev.line) {
    top = prev.top;
  } else {
    const ratio = (line - prev.line) / (next.line - prev.line);
    top = prev.top + ratio * (next.top - prev.top);
  }
  lockUntil = Date.now() + LOCK_MS;
  previewScroller.scrollTop = top;
};
```

### 10.1 第一步：拿到所有锚点

```ts
const list = anchors();
```

例如：

```ts
[
  { line: 0, top: 0 },
  { line: 5, top: 120 },
  { line: 12, top: 360 },
  { line: 20, top: 700 },
]
```

### 10.2 第二步：拿编辑器顶部行号

```ts
const line = getEditorTopLine();
```

比如用户现在滚到 Markdown 第 8 行：

```ts
line = 8
```

### 10.3 第三步：找前后两个锚点

循环逻辑：

```ts
for (let i = 0; i < list.length; i++) {
  if (list[i].line <= line) {
    prev = list[i];
    next = list[i + 1] ?? list[i];
  } else {
    break;
  }
}
```

如果当前行是 8，锚点是：

```ts
{ line: 5, top: 120 }
{ line: 12, top: 360 }
```

那么：

```ts
prev = { line: 5, top: 120 }
next = { line: 12, top: 360 }
```

意思是：第 8 行在第 5 行和第 12 行之间。

### 10.4 第四步：线性插值

代码：

```ts
const ratio = (line - prev.line) / (next.line - prev.line);
top = prev.top + ratio * (next.top - prev.top);
```

继续刚才的例子：

```text
当前行 line = 8
prev.line = 5
next.line = 12
prev.top = 120
next.top = 360
```

计算：

```text
ratio = (8 - 5) / (12 - 5)
      = 3 / 7
      ≈ 0.43
```

```text
top = 120 + 0.43 * (360 - 120)
    ≈ 223
```

于是预览区滚到：

```ts
previewScroller.scrollTop = 223;
```

这就是“按行号在两个预览锚点之间估算位置”。

### 10.5 为什么不用百分比同步？

因为 Markdown 和 HTML 高度不一致。

例如：

```md
![大图](...)
```

Markdown 里只有一行，但预览里可能高度 600px。

如果只按滚动百分比同步，遇到图片、表格、代码块时会明显错位。

当前方案虽然也不是绝对精确，但比百分比同步更贴近内容位置。

## 11. 预览滚动 → 编辑器滚动

函数：`syncPreviewToEditor()`

代码：

```ts
const syncPreviewToEditor = () => {
  const list = anchors();
  if (list.length === 0) {
    return;
  }
  const st = previewScroller.scrollTop;
  let prev = list[0];
  let next = list[list.length - 1];
  for (let i = 0; i < list.length; i++) {
    if (list[i].top <= st) {
      prev = list[i];
      next = list[i + 1] ?? list[i];
    } else {
      break;
    }
  }
  let line: number;
  if (next.top === prev.top) {
    line = prev.line;
  } else {
    const ratio = (st - prev.top) / (next.top - prev.top);
    line = prev.line + ratio * (next.line - prev.line);
  }
  lockUntil = Date.now() + LOCK_MS;
  scrollEditorToLine(Math.round(line));
};
```

这和“编辑器 → 预览”方向是反过来的。

### 11.1 输入变了

编辑器方向的输入是：

```text
源码行号 line
```

预览方向的输入是：

```text
预览区 scrollTop
```

代码：

```ts
const st = previewScroller.scrollTop;
```

### 11.2 根据 top 找前后锚点

```ts
if (list[i].top <= st) {
  prev = list[i];
  next = list[i + 1] ?? list[i];
}
```

如果当前预览滚动位置是：

```ts
st = 223
```

锚点是：

```ts
{ line: 5, top: 120 }
{ line: 12, top: 360 }
```

那么当前预览位置就在这两个锚点之间。

### 11.3 反向插值出源码行号

```ts
const ratio = (st - prev.top) / (next.top - prev.top);
line = prev.line + ratio * (next.line - prev.line);
```

计算：

```text
ratio = (223 - 120) / (360 - 120)
      ≈ 0.43
```

```text
line = 5 + 0.43 * (12 - 5)
     ≈ 8
```

最后：

```ts
scrollEditorToLine(Math.round(line));
```

把编辑器滚到第 8 行附近。

## 12. 事件监听如何绑定

代码：

```ts
const onEditorScroll = () => {
  if (Date.now() < lockUntil) {
    return;
  }
  cancelAnimationFrame(rafId);
  rafId = requestAnimationFrame(syncEditorToPreview);
};
```

编辑器滚动时：

1. 如果还在锁定时间内，直接忽略。
2. 取消上一帧还没执行的同步任务。
3. 下一帧执行 `syncEditorToPreview()`。

为什么要用 `requestAnimationFrame`？

因为 scroll 事件触发频率很高。用 rAF 可以把同步计算限制到浏览器下一帧，避免滚动时过度计算。

预览区同理：

```ts
const onPreviewScroll = () => {
  if (Date.now() < lockUntil) {
    return;
  }
  cancelAnimationFrame(rafId);
  rafId = requestAnimationFrame(syncPreviewToEditor);
};
```

最后绑定事件：

```ts
editorScroller.addEventListener("scroll", onEditorScroll, {passive: true});
previewScroller.addEventListener("scroll", onPreviewScroll, {passive: true});
```

`passive: true` 表示监听器不会调用 `preventDefault()`，浏览器可以更流畅地处理滚动。

## 13. 清理逻辑

`createScrollSync()` 返回：

```ts
return {
  destroy() {
    cancelAnimationFrame(rafId);
    editorScroller.removeEventListener("scroll", onEditorScroll);
    previewScroller.removeEventListener("scroll", onPreviewScroll);
  },
};
```

这个 `destroy()` 会在 `App.tsx` 的 cleanup 中调用。

作用是：

- 取消未执行的 rAF。
- 移除编辑器滚动监听。
- 移除预览区滚动监听。

这是 React 组件里处理 DOM 事件时必须注意的点。

```text
用户滚动右侧 Preview
    ↓
previewScroller 触发 scroll
    ↓
onPreviewScroll
    ↓
检查 lockUntil，没锁才继续
    ↓
requestAnimationFrame(syncPreviewToEditor)
    ↓
读取 previewScroller.scrollTop
    ↓
anchors() 收集预览区 data-line 锚点
    ↓
找到 scrollTop 前后的两个锚点 prev / next
    ↓
根据 top 比例反推出源码行号 line
    ↓
设置 lockUntil
    ↓
scrollEditorToLine(Math.round(line))
```

## 15. 为什么这个实现比较适合当前项目

这个项目是 Markdown 编辑器，不是普通左右两个列表。

Markdown 和预览 HTML 的高度关系非常不稳定：

| Markdown 内容 | 预览高度特点 |
| --- | --- |
| 普通段落 | 高度较稳定 |
| 标题 | 高度与样式有关 |
| 图片 | 一行源码可能对应很高预览 |
| 代码块 | 高度取决于代码行数 |
| 表格 | 高度取决于内容和列宽 |
| 数学公式 | 高度取决于渲染结果 |

所以用 `data-line` 锚点比直接百分比同步更合理。

## 16. 这个实现的局限

当前实现是“实用近似”，不是像素级精确。

可能不完全精准的情况：

1. 一个 Markdown 块非常高，比如超大图片。
2. 某些插件生成的 HTML 高度和源码行数不成比例。
3. 图片加载前后高度变化，锚点 `offsetTop` 会变化。
4. MathJax 或异步渲染内容改变高度后，短时间内同步可能有偏差。
5. 只给顶层块加 `data-line`，块内部无法更细粒度定位。

但优点是实现简单、稳定、性能可控。

## 17. 如果以后要优化，可以考虑什么

### 方案 1：图片加载后重新同步

图片加载完成后高度会变，可以监听预览区图片 `load` 事件，在图片加载后重新触发一次同步。

### 方案 2：增加锚点密度

现在只给 `level === 0` 的顶层块加 `data-line`。

如果想更精细，可以考虑给部分嵌套块也加行号，但要防止锚点过密。

### 方案 3：缓存 anchors

现在每次滚动都会调用：

```ts
previewScroller.querySelectorAll("[data-line]")
```

如果文档很长，可以考虑在内容变化后缓存锚点列表，滚动时直接使用缓存。

不过当前项目文档规模通常不会特别大，现有写法更简单。

### 方案 4：ResizeObserver

预览区高度变化时，比如图片加载、窗口变化、主题切换，可以用 `ResizeObserver` 标记锚点需要重新计算。

### 方案 5：只允许单向同步

如果双向同步让用户觉得跳动，也可以只保留“编辑器 → 预览”。

当前项目选择双向，是为了预览区浏览时也能定位回源码。

| 函数 | 职责 |
| --- | --- |

| `destroy()` | 清理监听器 |

## 19. 最核心的一句话

> 当前同步滚动的本质是：MarkdownIt 给预览 HTML 注入源码行号 `data-line`，滚动时在源码行号和预览像素位置之间做线性插值，从而实现编辑器和预览区的双向联动。
