# 发布前图片校验与确认发布设计

**日期：** 2026-07-13  
**状态：** 已获用户批准

## 背景

发布草稿前，VellumStyle 使用 `scanMarkdownMedia()` 扫描 Markdown，再由 `findUnuploadedImages()` 判断正文是否包含本地图片或非微信素材域名图片。当前扫描器直接对完整源码运行正则，会把行内代码、围栏代码块等示例中的图片语法误判成真实图片。例如：

```markdown
最终统一为 `![imgDescription](imgUrl)` 语法。
```

与此同时，真实图片可能位于多级列表的缩进内容中，因此不能简单忽略所有缩进行。

## 目标

1. 代码示例中的媒体语法不参与上传校验。
2. 多级列表中缩进的真实图片继续参与扫描。
3. 发布警告列出每张问题图片的行号、类型和地址。
4. 用户理解风险后，可以仅对本次操作选择继续发布。
5. 不改变 Markdown 原文和最终文章显示。

## 非目标

- 不自动上传、替换或删除问题图片。
- 不永久关闭图片校验。
- 不增加“以后不再提醒”。
- 不改变微信发布接口。
- 不把普通四空格缩进内容统一视为代码并排除。

## 扫描设计

### 忽略范围

扫描媒体前，在原始 Markdown 中识别以下明确的代码范围：

- 行内代码，包括不同长度的反引号定界符；
- 反引号围栏代码块；
- 波浪线围栏代码块；
- HTML `<code>...</code>`；
- HTML `<pre>...</pre>`。

普通缩进内容不加入忽略范围，确保多级列表中的真实图片不会漏检。

### 保持源码坐标

不得通过删除代码文本或生成压缩副本后扫描，因为 `MediaRef.start/end` 还用于导入替换。实现应保留原始字符串，生成半开区间 `[start, end)`，媒体扫描完成后过滤与忽略范围相交的结果。

```ts
interface SourceRange {
  start: number;
  end: number;
}
```

过滤应检查范围相交，而不是只判断起点，以兼容 URL 范围和整段 HTML 标签范围两类 `MediaRef`。

### 围栏与行内代码边界

- 围栏结束符必须使用相同字符，且长度不少于起始围栏。
- 未闭合围栏从起始位置忽略到文末。
- 围栏内部不再解析行内代码。
- 行内代码按 Markdown 反引号串长度配对。
- HTML `code/pre` 匹配不区分大小写，并覆盖跨行内容。

## 问题图片数据结构

保留 `findUnuploadedImages()` 作为发布校验入口，但将返回值升级为结构化结果：

```ts
interface UnuploadedImage {
  url: string;
  line: number;
  column: number;
  sourceType: MediaSourceType;
  syntax: MediaSyntax;
  reason: "local" | "external" | "temporary" | "unsupported";
}
```

行列号以 1 为起点，由原始 Markdown 和 `MediaRef.start` 计算。发布 UI 至少显示行号、可读类型和 URL；列号可保留供定位或后续编辑器跳转使用。

校验涵盖：

- 本地路径；
- 非 `mmbiz.qpic.cn` / `mmbiz.qlogo.cn` 的远程图片；
- `data:`、`blob:` 等临时来源；
- 不支持的协议。

微信素材域名图片继续视为已上传。

## 发布交互

检测到问题图片后，不再只显示数量 Toast，也不立即终止整个发布意图，而是显示确认对话框：

- 标题：`检测到可能尚未上传的图片`
- 说明：继续发布后图片可能无法显示或以后失效。
- 列表项：`第 N 行 · 类型：URL`
- 默认操作：`返回检查`
- 危险操作：`仍然发布`

选择“返回检查”关闭提示并停留在发布窗口。选择“仍然发布”只跳过当前点击产生的这一次图片校验，随后继续执行标题、封面、公式等待、HTML 生成和草稿发布流程。下一次点击发布仍重新校验并提示。

确认状态不写入持久化设置，不自动修改正文。

## 组件与数据流

1. `scanMarkdownMedia(markdown)` 计算代码忽略范围，扫描并过滤媒体引用。
2. `findUnuploadedImages(markdown)` 将问题引用转换为结构化诊断。
3. `PublishDialog` 点击发布时执行校验。
4. 无问题时直接进入原发布流程。
5. 有问题时保存本次诊断并打开确认对话框。
6. 用户选择“仍然发布”后调用抽取出的实际发布流程，避免通过递归或持久标志绕过后续校验。

确认对话框可作为 `PublishDialog` 内部状态实现；若现有对话框基础组件适合复用，应遵循现有样式和焦点管理，不引入新的 UI 依赖。

## 错误处理

- 超长 URL 在界面中换行或截断，但应通过标题或可复制文本保留完整值。
- 同一 URL 在不同位置出现时按位置分别列出，便于逐处修复。
- 无法识别的协议显示为“不支持的图片地址”。
- 用户强制发布后，微信接口错误仍走现有错误处理，不与校验警告混合。

## 测试

### 扫描器

- 忽略行内代码中的 Markdown 图片、HTML 图片、Obsidian 图片和视频链接。
- 支持多反引号行内代码。
- 忽略反引号和波浪线围栏代码块。
- 忽略 HTML `code/pre` 区域。
- 未闭合代码围栏安全忽略到文末。
- 仍识别行内代码外的真实图片。
- 仍识别多级列表中的缩进图片。
- 保持扫描结果 `start/end` 对应原文位置。

### 发布校验

- 用户文章中的两个代码示例不再报告。
- 微信素材图片不报告。
- 本地、外部、临时和不支持来源返回正确原因及 1-based 行列号。
- 同 URL 多处出现时保留各自位置。

### 发布 UI

- 无问题图片时直接进入发布流程。
- 有问题图片时显示详情而非仅显示数量。
- “返回检查”不调用发布接口。
- “仍然发布”调用一次发布接口。
- 再次点击发布仍会重新提示。
- 标题、封面及现有发布错误行为保持不变。

## 验收标准

自动化验收以文末“可复现验收与测试夹具”定义的仓库内最小 fixture 为唯一基准，不依赖 AppData 中的外部文章。原始《公众号排版工具VellumStyle v1.7.0发布！UI重大优化.md》仅作为可选手工回归：不修改其中两处代码示例时应无误报，列表缩进微信图片仍能识别。
## 评审补充：语法边界与兼容性约束

### 与 Markdown 渲染语义保持一致

代码范围识别必须与项目现用 `markdown-it` 的语义对齐，而不是将源码中的任意反引号串当作围栏：

- 围栏起始必须出现在允许的块级容器位置；支持普通正文、引用容器和嵌套列表容器中的围栏。
- 围栏结束符必须与起始符使用相同字符，长度不少于起始符，且除容器前缀、允许的缩进和行尾空白外不得包含其他文本。
- 普通文本中出现的三个或更多反引号不自动成为围栏。
- 未闭合的块级围栏按 Markdown 语义延伸到文末。
- 行内代码按相同长度的反引号串配对，遵循 `markdown-it` 对转义反引号、首尾空白规范化和跨行 code span 的行为。
- 未找到匹配结束符的行内反引号视为普通文本，不能把其后的真实图片全部排除。
- 围栏内部不二次解析行内代码。
- HTML `<code>`/`<pre>` 支持属性、大小写和跨行内容；闭合范围可以合并重叠区间。未闭合标签按上下文分别处理：块级未闭合 `<pre>` 与 `markdown-it` HTML block 语义一致，忽略到文末；未闭合的行内 `<code>` 不停止后续 Markdown 解析，不排除其后真实图片。两种情况分别设置回归测试。
- 四空格缩进本身永远不是本功能的排除依据；是否处于围栏由块级容器状态判断。

新增边界测试：普通句子中的三反引号、带尾随文本的无效闭合围栏、未配对行内反引号后紧跟真实图片、引用与嵌套列表中的围栏、转义反引号、跨行 code span、带属性及未闭合的 HTML code/pre。

### 完整问题分类

`findUnuploadedImages()` 必须覆盖所有 `MediaSourceType`：

| sourceType | reason | UI 类型标签 |
|---|---|---|
| `local` | `local` | 本地图片 |
| 有效的非微信 `remote` | `external` | 外部图片 |
| `data` / `blob` | `temporary` | 临时图片 |
| `anchor` / `empty` / `unsupported` | `unsupported` | 不支持的图片地址 |
| 解析失败的远程地址 | `unsupported` | 不支持的图片地址 |

UI 类型标签只由 `reason` 映射，`syntax` 用于内部诊断而不混作风险类型。空地址显示为 `（空地址）`。

微信素材地址必须在补全 `//` 为 `https:` 后由 `URL` 成功解析，并且 hostname 精确等于 `mmbiz.qpic.cn` 或 `mmbiz.qlogo.cn`。删除当前基于字符串 `includes` 的兜底，防止畸形或伪造地址被判定为已上传。

### 确认界面的具体实现

警告不叠加第二个同层 `Dialog`。`PublishDialog` 使用内部视图状态，将原内容区和页脚临时切换为警告详情视图，从而复用同一模态层和现有关闭行为。

- 警告视图标题和说明需具有可访问关联；进入时焦点放在“返回检查”。
- Escape 等价于“返回检查”，返回后恢复到发布按钮或触发前焦点。
- 父发布窗口关闭、重新打开及任一操作完成后清空警告诊断和确认状态。
- 点击“仍然发布”后立即进入 `busy` 并禁用两个操作；使用同步重入守卫防止快速双击导致 `addDraft()` 多次调用。
- 抽取的实际发布函数必须完整保留现有标题、封面、发布设置持久化、`busy`、`pubResult`、MathJax 等待、HTML 生成、成功延时及错误处理。
- “仍然发布”只授权当前诊断快照对应的一次执行；正文变化或下次点击发布必须重新扫描。

### 全局消费者兼容性

`scanMarkdownMedia()` 的行为变化同时影响 Markdown 导入、上传替换、封面候选和发布校验，因此必须补充：

- 代码示例不计入导入数量，不触发上传，不参与替换。
- 非代码媒体仍保持 URL 精确坐标；HTML 图片和 Obsidian 嵌入仍保持原有整 token 替换坐标。
- 只出现在代码中的微信图片不进入封面候选，真实微信图片仍进入。
- 非代码媒体的现有导入替换行为及缩放元数据保持不变。

## 可复现验收与测试夹具

在测试中嵌入一个最小但与问题文章对应的固定 Markdown fixture，不依赖用户 AppData 文件：包含两处原样行内示例、至少一张多级列表缩进微信图片和一张可选本地问题图片。fixture 明确断言：

- 仅含两处代码示例和微信图时，问题数量为 0，微信封面候选数量符合预期。
- 加入本地图片的固定行号后，诊断数量为 1，并精确匹配该行号、URL、`sourceType` 和 `reason`。
- 渲染后的代码 span 文本仍为 `![imgDescription](imgUrl)` 与 `![imgDescription](imgUrl =缩放参数)`，不包含额外反斜杠。
- 传入草稿发布命令的 HTML 保留上述可见示例文本。
- 使用 mock Tauri `add_draft` 验证“返回检查”为 0 次调用、“仍然发布”为恰好 1 次调用，包含快速重复点击场景。

真实微信草稿创建仅作为需要有效凭证的手工集成检查，不作为自动化测试通过条件。

### 警告视图交互测试补充

- 打开警告视图后，初始焦点位于“返回检查”。
- 按 Escape 只退出警告视图，不关闭父发布窗口，`add_draft` 为 0 次，并将焦点恢复到发布触发按钮。
- 关闭并重新打开 `PublishDialog` 后，不保留旧诊断、一次性授权或警告视图状态。
- 警告出现后若正文快照变化，必须重新扫描；完成一次返回或发布后再次点击发布也必须重新扫描，不能复用上次确认。
